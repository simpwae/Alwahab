**Three real, pre-existing bugs surfaced during the port** (all silently present in the original Vite app too, just never caught because `vite build` never runs `tsc`, and this migration's stricter Next.js/TypeScript build does):
1. Roughly two dozen files reference the bare type `ReactNode` (or `FormEvent`, `ButtonHTMLAttributes`, etc.) without importing it from `react` — works under Vite's untyped esbuild transform, hard TypeScript error under `next build`. Fixed by adding the missing named imports everywhere it appeared (no logic changes).

Two things needed actual design work, not just mechanical porting:
- **`ProtectedRoute` needed an `isInitialized` flag added to `AuthContext`.** Wave 1's SSR-safety fix means `user` always starts `null` on every render (real auth state only arrives after a post-mount `useEffect`). Without a way to distinguish "still checking" from "confirmed logged out," a naive port of `ProtectedRoute` would redirect every visitor to `/login` for a frame before the real session loads. `isInitialized` (false until that first effect resolves) fixes it: `ProtectedRoute` renders nothing until initialized, then decides.
- **Logging out from a protected page raced two navigations and lost.** `AccountNav`'s logout button calls `logout()` then navigates home; `ProtectedRoute`'s effect *also* reacts to `user` becoming `null` and redirects to `/login`. In the old react-router app this was never a problem (`navigate()` swaps the rendered route tree synchronously, so the old page's `ProtectedRoute` is already gone by the time state settles). Next.js's App Router navigation is transition-based, not synchronous, so both redirects were live at once — and `/login` was winning, landing users on the login page immediately after logging out instead of on the homepage. Fixed by making logout a hard navigation (`window.location.href = '/'` after `logout()`, which synchronously clears `localStorage` before the reload) — sidesteps the race entirely and, as a bonus, guarantees every other context resets to its default state too, which is arguably what "log out" should mean anyway.
