The scaffold landed on **Next.js 16.2.10 / React 19.2 / Tailwind v4** (current `create-next-app` defaults) rather than matching the old app's Vite/React 18/Tailwind v3 versions exactly — Next 16 ships its own `AGENTS.md` warning that it has real breaking changes from older training-data assumptions, so its bundled docs (`node_modules/next/dist/docs/`) were read before writing App Router code rather than assumed. Consequences worth knowing:
- Tailwind v4 uses CSS-first config (`@theme` block in `globals.css`), not `tailwind.config.js` — the same color/radius/shadow/font tokens were re-declared there, not copy-pasted as a JS file.
- All 34+ ported files are Client Components (`"use client"`) for now, uniformly — no Server Component adoption in this wave, deliberately (stated in the plan as a later optimization, not a parity requirement).
- Router API swap throughout: `react-router-dom`'s `Link`/`useNavigate`/`useParams`/`useSearchParams` → `next/link`'s `Link`/`next/navigation`'s `useRouter`/`useParams`/`useSearchParams`. Pages using `useSearchParams` (`/search`, `/track-order`, `/category/[slug]`, `/deals`) are wrapped in `<Suspense>` at the route level, per Next's requirement.
- **`navigate(path, { state })` has no Next.js equivalent** — react-router's location-state passing had to be redesigned, not just swapped: `CartContext` gained an `appliedCode`/`setAppliedCode` field so the coupon applied on `/cart` survives the navigation to `/checkout` (Checkout now recomputes the discount itself from `appliedCode` + `subtotal` + `coupons` instead of trusting a passed-in number — arguably *more* correct than the original). The Checkout→OrderConfirmation state-passing was dropped entirely since `OrderConfirmation` already had a `getOrderById(orderId)` fallback that's sufficient on its own (the state was only ever a same-tick optimization).

**Three real, pre-existing bugs surfaced during the port** (all silently present in the original Vite app too, just never caught because `vite build` never runs `tsc`, and this migration's stricter Next.js/TypeScript build does):
1. Roughly two dozen files reference the bare type `ReactNode` (or `FormEvent`, `ButtonHTMLAttributes`, etc.) without importing it from `react` — works under Vite's untyped esbuild transform, hard TypeScript error under `next build`. Fixed by adding the missing named imports everywhere it appeared (no logic changes).
2. `Header.tsx`'s search-suggestions dropdown attaches `useRef<HTMLDivElement>` to a `<form>` element — a real type mismatch, caught by `next build`'s type-check. Fixed by correcting the ref's generic type to `HTMLFormElement`.
3. **`AuthContext`/`AdminAuthContext` read `localStorage` synchronously as their `useState` initializer.** Next.js server-renders Client Components too (for the initial HTML, both at build-time static generation and at request time) — `localStorage` doesn't exist during that server pass, so this crashed the build outright (`ReferenceError: localStorage is not defined`). Fixed by initializing both to `null`/`null` on first render (matches server and client exactly, avoiding a hydration mismatch) and moving the actual `localStorage` read into a `useEffect` that only fires post-mount in the browser. Same root cause independently caught a second time in `FlashDeals.tsx`'s countdown timer (`Date.now()` computed during the initial render differs between the server pass and the client hydration pass by design, producing a real "server text didn't match client" hydration-mismatch console error on `/`) — fixed the same way, deferring the `Date.now()`-dependent state to `useEffect`.

Verified with the same scratchpad Playwright setup as prior sessions: screenshotted all 13 routes side-by-side against the still-running old Vite app (visually identical, module the hero carousel's independent auto-rotate timer), then ran a full click-through flow on the new app only — browse → add to cart → apply a coupon on `/cart` → checkout → place a COD order → land on `/order-confirmation/[id]` with the correct discount/total/address → click through to `/track-order` and find the same order by ID+email with the right status timeline. Zero console errors on the final run (the one hydration mismatch and one `localStorage` crash both surfaced *during* this verification pass and got fixed before it passed clean). `next build`, `npx eslint .`, and `npx vitest run` (ported the 3 relevant pure-function test files — `sampleCoupons.test.ts`, `productFilters.test.ts`, and a `Checkout` test now colocated as `app/checkout/page.test.ts`) are all clean; one new ESLint default (`react-hooks/set-state-in-effect`, part of `eslint-config-next` 16) was disabled repo-wide in `next-app/eslint.config.mjs` with an explanatory comment, since it blanket-flags the "reset state when a route param changes" and "read a browser API once after mount" patterns used throughout — both legitimate here, not bugs.
